# 邮件自动处理机器人落地指南（面向 GitHub + Codex 新手）

> 目标：用**尽可能低门槛**的方法，先把“收件→分类→分发→摘要→归档→周报”跑起来，再逐步升级到更高自动化。

---

## 1. 你要先统一的业务规则（先规则，后自动化）

在写任何代码前，请先把以下“标准”固定，否则机器人会越做越乱。

### 1.1 标签体系（你已经有了雏形）

- 事务前缀：`GOV / COM / FIN / OPS / CUS / INT`
- 优先级：`P0 / P1 / P2`
- 状态：`Action Needed / Waiting External / Closed`

建议组合成统一标签，例如：

- `GOV-P0-Action Needed`
- `FIN-P1-Waiting External`

### 1.2 标准邮件标题格式（转发/升级用）

```text
[前缀][优先级][状态] 原主题 | Owner:姓名 | Due:YYYY-MM-DD
```

例如：

```text
[GOV][P0][Action Needed] VAT Filing Notice | Owner:Li Lei | Due:2026-03-10
```

### 1.3 三行摘要模板（强制）

```text
背景：这封邮件来自（机构/供应商），涉及（事项）。
要点：1) ... 2) ... 3) ...
需要动作：请（负责人）在（截止日期）前完成（动作）。
```

### 1.4 SLA（处理时限）建议

- `P0`：2小时内响应、24小时内给出处理方案
- `P1`：24小时内响应、3个工作日内推进
- `P2`：48小时内响应、按周节奏处理

## 3. 分阶段实施路线（建议 4 周）

## 第 0 周：准备清单

- 建立 `路由矩阵`（谁是 Must 收件人 / 备份人）
- 建立 `标签字典`（每个前缀适用范围）
- 建立 `邮件模板`（三行摘要、催办模板、回读确认模板）
- 建立 `共享盘目录和命名规则`

共享盘命名建议：

```text
/YYYY/MM/[类别]/YYYYMMDD_类别_主题_对方_v1.pdf
```

## 第 1 周：先实现“收件-分类-分发-提醒”（MVP）

流程（自动触发：收到新邮件）：

1. 抓取邮件（主题、正文、发件域名、附件）
2. 调用 AI 分类（输出：prefix、priority、phishing_risk、summary）
3. 根据 `prefix + priority` 查路由矩阵
4. 自动转发/抄送 Must 人员
5. 自动追加三行摘要
6. 对 `P0/P1` 自动创建提醒任务（飞书/Outlook Task）

## 第 2 周：加入“归档 + 附件入库 + 周报”

1. 自动打标签并移动到文件夹
2. 附件保存到共享盘（按命名规则）
3. 每周五自动汇总：
   - 政府/监管来信列表
   - 账单与付款到期
   - 未闭环事项
   - 风险提示

## 第 3-4 周：增强“可追踪”

1. 增加处理日志表（谁收到、何时处理、交给谁、下一步）
2. 超 SLA 自动催办并记录催办次数
3. 月度抽样审计 10-20 封关键邮件

## 7. 你最关心的两个痛点，如何被覆盖

### 痛点 1：马耳他重要邮件没同步中国

用“`GOV/FIN` 自动强制抄送 Must 列表 + P0/P1提醒 + 日志表”解决：

- 任何 GOV/FIN 自动抄送中国总经理 + 对应负责人
- 自动创建任务和截止日期
- 周报自动列出“未闭环”

### 痛点 2：口述指令多次转述造成误解

用“**三行摘要 + 回读确认模板**”解决：

- 转发必须有背景/要点/动作
- 收件人回复固定格式：
  - 我理解的任务：...
  - 我计划完成时间：...
  - 风险/阻塞：...

这样可以把口头信息变成可追踪文本。

## 8. 安全与合规底线（必须做）

- 只让机器人访问“共享邮箱”而非个人邮箱
- API Key 放环境变量，不写死在代码
- 对含个人隐私/财务信息的邮件做最小必要处理
- 对疑似钓鱼邮件设置隔离流程，不自动点链接
- 记录机器人决策日志，便于审计

## 11. 你问的方案：Python + 飞书 Webhook + OpenAI API，能不能上线？

- `requests`（调用飞书和 OpenAI API）

## 12. 你新问的方案：能不能用 WPS 替代飞书 Webhook？用多维表格当数据库可行吗？

答案：**可行，但建议“WPS 主数据 + 通知通道兜底”**，不要只依赖单一通知入口。

### 12.1 结论先说

- **WPS 多维表格当轻量数据库：可行**（适合 MVP 和中小规模流程）
- **WPS 完全替代飞书 Webhook：部分可行**（取决于你是否需要强提醒、@人、回执与机器人交互能力）
- 如果你想“稳上线”：建议采用
  - `WPS 多维表格（主记录） + 企业微信/邮件通知（兜底） + Python 智能体（编排）`

### 12.2 什么场景下适合用 WPS 多维表格

适合：

- 你们团队已经日常使用 WPS
- 先追求“可追踪、可审计、可看板”，不是极限实时
- 每日邮件量中等（例如几十到几百）

不太适合：

- 高频实时预警（分钟级、秒级）
- 复杂多系统联动（大量审批、跨系统写回）
- 对机器人交互和消息回执依赖很重

### 12.3 推荐架构（你可以直接落地）

```text
共享邮箱/邮件API -> OpenClaw风格智能体编排层 -> OpenAI分类/摘要 -> WPS多维表格(主库)
                                                      -> 通知通道(飞书/企业微信/邮件三选一)
                                                      -> 附件归档(WPS云文档或共享盘)
```

WPS 多维表格可作为“邮件工单总台”，关键字段建议：

- message_id（唯一键）
- received_at
- sender
- prefix（GOV/COM/FIN/OPS/CUS/INT）
- priority（P0/P1/P2）
- status（Action Needed / Waiting External / Closed）
- owner
- due_at
- must_sync_cn（是否必须同步中国）
- summary_background / summary_points / summary_action
- reminder_count
- archive_link

### 12.4 用“智能体”时，建议你这样分工

- **规则引擎（确定性）**：
  - 路由矩阵、Must抄送、SLA计时、命名规则
- **LLM 智能体（非确定性）**：
  - 分类建议、三行摘要、双语翻译、周报草稿

原则：

- 涉及合规/财务付款的最终动作必须可人工确认
- 智能体负责“建议与草稿”，规则引擎负责“硬约束”

### 12.5 你最关心的可追踪，如何在 WPS 里实现

在多维表格中增加 4 个追踪字段：

- assigned_to（交付给谁）
- assigned_at（何时交付）
- next_action（下一步）
- last_action_by（最后操作者）

再增加 2 个自动视图：

- `P0/P1 超时视图`
- `GOV/FIN 未同步中国视图`

这样就能直观看到“谁收到、何时处理、交付给谁、下一步是什么”。

### 12.6 给你的实际建议（基于你当前基础）

你现在可走这条最稳路径：

1. Python 先打通：读取邮件样本 -> 分类 -> 写入 WPS 多维表格
2. 再加通知：P0/P1 同时发“邮件提醒”给负责人（后续再接飞书/企微）
3. 再加周报：每周五自动从多维表格生成周报草稿
4. 最后再做闭环：催办次数、超时升级给总经理

这条路径对 Python 初学者非常友好，且能在 1-2 周内做出可演示版本。

## 13. 升级为 OpenClaw 风格智能体：分步骤实战（我会这样带你做）

下面是我们从 0 到上线的执行路径，每一步都写明“作用”。

**你要做什么**

### 第 2 步：搭建 OpenClaw 风格“编排层”骨架（2-3 天）

**你要做什么**

- 建一个 `agent_orchestrator`（编排器）
- 拆成 5 个可插拔模块：
  1. `ingest`（收件）
  2. `classify`（分类）
  3. `route`（分发）
  4. `archive`（归档）
  5. `report`（周报）

**作用**

- 这就是 OpenClaw 风格核心：**一个编排器 + 多个能力工具**。

### 第 3 步：先做“离线回放”而非直连生产邮箱（2 天）
